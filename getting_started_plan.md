## How to use this plan

- Work through milestones in numeric order unless the user explicitly reprioritizes them.
- Do not begin Milestone 2 until the Milestone 1 hard feasibility gate is checked complete.
- A task may be checked only after its implementation, tests, and stated completion evidence are finished.
- When completing a task, change **[ ]** to **[x]** and append: **— Completed: YYYY-MM-DD HH:mm:ss +HH:MM (Asia/Jerusalem)**.
- Check a milestone only after every required task beneath it and its exit criteria are complete; append the same timestamp format.
- If a task is blocked, leave it unchecked and add an indented **Blocked:** note with evidence, attempted approaches, and the next decision required.
- If implementation reveals missing work, add a new numbered checkbox under the correct milestone before doing that work.
- Keep CODEX_INTER_AGENT_MESSAGING_BRIDGE.md as the architecture contract. Record any deliberate architectural change in both files before implementing it.

## Non-negotiable project constraints

- Messaging is invoked on demand by an agent tool call; there is no coordinator thread or autonomous orchestration application.
- Every participating thread runs through the bridge-managed, capability-token-authenticated shared app-server; arbitrary independently owned desktop sessions are outside the MVP boundary.
- Existing Codex threads use MCP as the primary tool exposure path.
- Normal delivery uses a real app-server turn/start on the recipient thread.
- Sender identity comes from trusted host or process context, never model-supplied arguments or a thread title.
- Thread titles may assist operator discovery but are never persistent routing identities.
- Busy recipient threads queue normal messages; normal delivery never silently uses turn/steer.
- Only the authoritative final agent message is returned; reasoning, intermediate commentary, command output, approvals, and hidden chain-of-thought are not forwarded.
- The implementation must not automate the Codex UI or directly mutate Codex session, rollout, or internal SQLite files.
- Exact app-server fields come from schemas generated by the installed Codex binary.
- Side-effect approvals are never automatically granted merely because another agent requested them.
