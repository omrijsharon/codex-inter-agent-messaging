# Threat Model

This review covers synchronous, asynchronous, and group messaging plus the plugin-provided automatic-bootstrap path in which participating Codex clients and caller-bound MCP processes connect to one bridge-managed app-server. The bridge is transport infrastructure, not a coordinator, and it starts agent work only after an explicit tool call or while recovering a previously accepted unfinished delivery.

## Assets and trust boundaries

- The stable agent registry, per-generation owner bindings, messages, attempts, leases, inbox state, groups, membership snapshots, ACL rules, and audit events are trusted local state.
- `BRIDGE_AGENT_ID`, the database path, app-server endpoint, and capability-token path are trusted process configuration. The model cannot supply or override them.
- MCP arguments and all peer message bodies are untrusted model-controlled input.
- The app-server is trusted only after capability-token authentication. Loopback is the default; remote transport requires explicit opt-in, `wss:`, normal certificate and server-name validation, and an operator-managed CA when applicable.
- The administrative CLI is a local operator surface. Its authority is the current OS account's access to the bridge data directory and app-server capability token; it is not exposed as an MCP tool.
- The installed plugin, its `.mcp.json`, the package executable path, and trusted caller-identity configuration are executable local authority. Installation or modification grants code execution as the current OS user and must come from a reviewed source.
- The bootstrap lock and connection descriptor are untrusted hints until an owner-only capability token authenticates a health response that matches the expected host nonce, version, installation identity, database identity, and owner mode.
- Official desktop and IDE app-server children are separate ownership domains. Tool discovery does not authorize the bridge to operate on their privately owned threads.

## Threats and controls

| Threat                                        | Control                                                                                                                                                                               | Residual risk                                                                                                                                 |
| --------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| Sender or destination spoofing                | Sender is bound by `BRIDGE_AGENT_ID`; MCP schemas reject extra metadata; recipients resolve only through stable registry IDs.                                                         | A compromised MCP process configuration can impersonate the agent it is configured for.                                                       |
| Prompt injection or envelope escape           | Authenticated metadata is generated by the bridge; peer text is JSON encoded and byte-length framed as untrusted content.                                                             | A recipient model may still follow malicious instructions contained in the clearly marked peer body.                                          |
| Unauthorized peer access                      | Durable pairwise ACL rules support explicit allow/deny and configurable default denial; unavailable generations are rejected.                                                         | Default allow remains available for simple trusted-local deployments and must be chosen consciously.                                          |
| Replay or duplicate delivery                  | Sender-scoped idempotency keys, unique message IDs, `clientUserMessageId`, history reconciliation, and FIFO leases suppress normal duplicates.                                        | A crash at an app-server boundary with incomplete history could still require operator reconciliation.                                        |
| Synchronous deadlock                          | Active dependency edges and authenticated call chains reject ancestor cycles; TTL and retry bounds terminate stalled work.                                                            | Semantically circular work split across unrelated conversations cannot always be inferred.                                                    |
| Denial of service                             | Message bytes, queue depth, call-chain length, hop count, TTL, retry count, and local concurrent-delivery limits are bounded.                                                         | Multiple compromised local MCP processes can consume CPU, disk, or app-server capacity within their separate process limits.                  |
| Asynchronous reply loop                       | Recipient assistant output is never routed automatically; every reply requires a later authenticated `reply_to_message` call and becomes one audited edge.                            | Agents can still explicitly create a long exchange until TTL, queue, or operator controls intervene.                                          |
| Inbox disclosure                              | Inbox reads, acknowledgements, status queries, and replies are bound to the trusted caller identity and reject messages outside its sender/recipient scope.                           | The trusted local database owner can read all stored message bodies directly.                                                                 |
| Group membership or visibility leak           | Group listing and message status require current membership or immutable snapshot visibility; membership administration is local CLI-only.                                            | Historical snapshot members retain visibility of messages they were authorized to receive at send time.                                       |
| Group fan-out exhaustion                      | Active membership is snapshotted, fan-out and per-recipient queues are bounded, pairwise ACL/readiness checks run before creation, and each delivery is independently limited.        | Multiple allowed group sends can still grow the local database and consume bounded delivery capacity.                                         |
| Unsafe selective retry                        | Only the original sender can retry; only failed/dead-letter snapshot recipients are eligible; current identity, ACL, readiness, and queue limits are re-checked.                      | A recipient removed after the original send remains a historical snapshot target if current pairwise authorization still permits the retry.   |
| Unauthorized synthesis                        | Gathering is visible only to snapshot participants, callable only by the original sender, includes only explicit replies, and names the synthesizing agent.                           | The synthesizing model may summarize explicit replies incorrectly; the bridge does not validate semantic fidelity.                            |
| Approval escalation                           | Recipient turns use `approvalPolicy: never`; command and file approvals and MCP elicitations are explicitly declined; unsupported permission requests receive an error.               | A future app-server approval method must be added to the installed-schema policy before it can be safely supported.                           |
| Secret or content leakage                     | Default structured logs redact credential/content/path keys and sensitive value patterns; persisted delivery errors sanitize detected sensitive text; peer replies are sender-scoped. | Novel secret formats under neutral keys may evade pattern redaction. Operators must avoid debug logging of raw protocol frames.               |
| Stale or wrong thread delivery                | Generation foreign keys, stable installation/database/protocol bindings, exact resumed/read thread IDs, and verified replacement/adoption prevent silent rebinding.                   | A malicious app-server with the valid capability token can lie; the public protocol cannot detect a private owner opened after adoption.      |
| Local database or token theft                 | Capability tokens are generated with high entropy and owner-only creation mode where supported; transport is authenticated. Data stays under the configured local account directory.  | Malware or another process running as the same OS user can read the token, database, and message content. At-rest encryption is not provided. |
| Audit tampering                               | SQLite triggers append message and attempt lifecycle events containing correlation IDs and timestamps.                                                                                | The same OS account that owns the database can directly modify it; the audit trail is not externally signed or append-only storage.           |
| Plugin or launch-config replacement           | Plugin provenance, manifest validation, relocatable packaged paths, version checks, and operator-controlled installation/configuration keep launch authority outside model arguments. | Malware or another process running as the same OS user can replace executable files or trusted configuration.                                 |
| Concurrent bootstrap split-brain              | Atomic exclusive startup lock, authenticated double-check after lock acquisition, bounded readiness, and incompatibility refusal converge callers on one owner.                       | Filesystems with broken local locking semantics are unsupported; same-user malware can interfere with lock files.                             |
| Stale descriptor, PID reuse, or port spoofing | Descriptor schema validation plus an authenticated health response matching a random host nonce and expected identities; PID/port checks alone are rejected.                          | Same-user token theft permits impersonation until the operator rotates state and restarts the host.                                           |
| Capability token disclosure                   | Token is stored separately from the descriptor with owner-only creation permissions, never placed in process arguments, plugin metadata, or normal logs, and is redacted from errors. | Windows ACL inheritance and same-user processes remain within the trust boundary; at-rest encryption is not provided.                         |
| Hidden child or orphaned process              | Detached startup closes MCP stdio, records host identity, uses bounded readiness, and exposes authenticated status/stop/restart operations.                                           | OS termination can leave stale metadata, which recovery must quarantine conservatively.                                                       |
| Version-skew split-brain                      | MCP and host negotiate package/protocol/owner-mode compatibility; an incompatible healthy host returns `HOST_INCOMPATIBLE` and blocks a second launch.                                | Upgrades may require an explicit operator stop/restart and temporary unavailability.                                                          |
| Desktop ownership confusion                   | Unbound/foreign generations fail before app-server calls; supported CLI targets use the authenticated remote wrapper; desktop/IDE targets return `UNSUPPORTED_THREAD_OWNER`.          | Users may see plugin tools in desktop, and no exclusive upstream owner claim can detect a later private owner on the pinned build.            |

## Approval and administrative policy

Unattended peer work never receives elevated authority. Command execution and file-change approval requests are declined, MCP elicitations are declined, and unsupported permission-elevation requests fail closed while the event reader remains responsive. Human approval routing is outside v0.4.

Administrative operations require direct execution by the local OS account that owns the bridge configuration and data directory. They are intentionally absent from the model-facing MCP server. Replacement and owner adoption require exact agent-ID confirmation and live thread verification; adoption additionally requires idle status.

## Automatic-bootstrap and lifecycle policy

The host is a per-user detached singleton with no automatic idle shutdown in the first plugin release. An MCP or Codex task exiting cannot terminate it. Authenticated operator commands handle status, stop, and restart; operating-system shutdown may terminate it normally. Startup state is fail closed, bounded, and contains no model-controlled command fragments.

The supported topology remains the bridge-managed shared owner. Plugin installation may expose tools in other Codex surfaces, but it does not make a private desktop- or IDE-owned thread safe for cross-thread delivery. No integration may compensate by patching a binary, automating the UI, scraping logs, or mutating Codex internal histories.

## Accepted residual risks

The project assumes the local OS account, installed plugin/package, Codex binary, app-server process, MCP configuration, trusted caller binding, and capability token are not compromised. It does not provide at-rest message encryption, tamper-evident external audit storage, certificate pinning beyond platform TLS validation, multi-user administrator roles, a Windows service manager, or safe execution of arbitrary peer instructions. Remote app-server transport is an expert configuration and is not part of the normal local deployment.
